# c4 热议标的温度计——喊单×实盘×内部人三格（周跑，可事件驱动加跑）

本 skill 仅供社群运营人员触发使用（群成员不直接交互），周跑，可事件驱动加跑——比如 c5 热点扫描判定某标的适合并回温度计时，可临时对该标的单独加跑一次，不必等到下周固定档期。产出单篇贴文，分两段：总览段是本周推特在吵什么（喊单聚合榜 + 多空比）；主体段是对喊单榜 Top 3-5 逐个给出温度计三格——🐦 推特情緒 / 💰 真金白銀 / 🏢 內部人動向三个维度交叉比对，跨源打架时明写、不硬圆。frontmatter description 已点明本 skill 的定位边界：只呈现温度、不给跟单指令——三格读数即使全部指向同一方向，也不能因此写出"所以应该跟上"这类结论句（S-6 禁令镜像，见第 2 节收尾）。

## 1. 调用序列（≈4-6/周，可事件驱动加跑）

调用序列：

| 步骤 | 调用 | 额度 |
|---|---|---|
| 1 | `signal(query="consensus", asset_type="tradfi", time_range="3d")` 无 categories 一次拿全四类（新 caveat N-4） | 1 |
| 2 | 对喊单榜 Top 3-5 逐个 `signal(query="<TICKER> 详细仓位", asset_type="tradfi", time_range="7d")` 四维钻取（实测 ticker 经 query 正确解析进 keywords；query 禁放"KOL"等元词） | 各 1 |
| 3 | （可选）`news(query="<TICKER> <公司名>", time_range="24h")` 补推特层原文 | 0（实测） |

步骤 1 的"全四类"对应 signal 工具本身的四个类别：kol_call（喊单）、trader_position（实盘持仓）、insider_trading（内部人+国会议员交易）、institutional（13F 机构持仓）——不带 categories 参数即为默认 fanout，一次拿全四类仍只计 1 额度（N-4）。步骤 1 的 time_range="3d" 只用于总览段的喊单聚合榜快照，步骤 2 对选中标的深挖时改用 7d 窗口，二者时间窗不同，不要混用。

frontmatter 的 mcp 声明含 `mcp__followin__metrics`，但上表三步核心序列本身不调用它——若运营想在温度计贴文里额外给某标的加一行即时价格/涨跌幅做视觉锚点，可选调用 `metrics(query="<TICKER>", asset_type="tradfi")` 补一次快照（不计入标称额度 ≈4-6/周，需另计 1 点）；没有这类展示需求时，三步核心序列完全不需要 metrics。

调用形态铁律（本序列全程通用）：

> **调用形态铁律（2026-07-20 起生效，trend-scout v1.11.1 实测 + 2026-07-22 复现）**：`keywords/categories/sources` 等数组参数在当前环境会被序列化成字符串遭 schema 拒（连环 -32602）。所有调用规范以 **query 自然语言/空格拼串为主写法**（服务端自解析成 keywords，meta 可验证），数组形式仅作"标准客户端若可传数组"的备选注记。批量降级梯：① keywords 数组批量（≤10，B-31）→ ② query 串批量（crypto 实测可行，tradfi 多 ticker 可能被路由到 fundamentals，实现时验证）→ ③ 单 ticker 并行、每批 ≤4 路（SSE 红线）。另：Followin session 每 5-8 次调用可能短挂，重试 1 次即恢复，还不行让运营 `/mcp restart followin`。

每步 query 主形态调用示例：

```
1. signal(query="consensus", asset_type="tradfi", time_range="3d")
   # 无 categories 一次 fanout 全四类（kol_call/trader_position/insider_trading/institutional），仍只计 1 额度（N-4）

2. signal(query="<TICKER> 详细仓位", asset_type="tradfi", time_range="7d")
   # 对步骤 1 喊单榜 Top 3-5 逐个钻取，温度计三格的素材皆出自这一步；query 禁放"KOL"等元词——会被解析成 crypto 关键词，实测空返（防坑镜像见第 3 节）
   # Top 5 时按 SSE 红线单批 ≤4 路，拆成 4+1 两批并行

3. news(query="<TICKER> <公司名>", time_range="24h")
   # 可选，补推特层喊单帖原文；quota=0（实测）；搜索模式不传 asset_type（红线 1）
```

## 2. 产出模板：温度计三格

spec 产出结构（逐字）：

> 产出结构：本周推特在吵什么（喊单聚合榜 + 多空比）→ 每个标的的温度计三格：
> - 推特情緒（喊单方向比 + A 级 KOL 代表观点，正反都放）
> - 真金白銀（实盘持仓方向/人数/杠杆——数据薄就明说薄）
> - 內部人動向（近 90 天 S-Sale/P-Purchase，剔除 F-InKind）
> → 跨源打架时明写结论（"情緒滿格 ≠ 大錢進場"）→ 名词卡。

落地结构：

1. **标题**：📌 + 本週熱議溫度計 + 日期区间
2. **一句話先懂**（≤40 字）：本周喊单最热的 1-2 个标的 + 整体多空比一句
3. **模块正文**，分两段：
   - **总览段**：本周推特在吵什么——喊单聚合榜（Top 3-5 标的排名）+ 整体多空比一行，素材来自步骤 1 consensus 返回的聚合结果
   - **主体段**：对总览榜单 Top 3-5 逐个给出温度计三格，每个标的固定三行：
     - 🐦 **推特情緒**：喊单多空比（如"多 8：空 2"）+ A 级 KOL 代表观点正反各一句，注明 tier（如"（A 級 KOL）"）——正反都要放，不能只挑一边
     - 💰 **真金白銀**：实盘持仓方向/人数/杠杆一句；统计前先过滤 tier 为空、notional 为 null 的行（第 3 节镜像）；标的属 stock_perp（代币化股票永续）时标注「鏈上股票永續交易員」而非「美股交易員」；过滤后样本仍太薄就直接写「本週實盤數據樣本太薄，不足以下判斷」，禁止硬凑（S-7 铁律 3 镜像）
     - 🏢 **內部人動向**：近 90 天窗口，客户端按 transactionDate 过滤（第 3 节 N-6 镜像），只认 S-Sale（卖出）、P-Purchase（买入）两种；90 天内无符合条件记录就写「近 90 天無內部人申報異動」
4. **接下來看什麼**（可选）：若某标的三格分歧巨大，提示下次跑温度计时留意哪个维度会不会收敛
5. **今日名詞卡**
6. **免责声明**（S-11 对外版逐字）："⚠️ 以上整理自公開研報與市場數據，僅做資訊分享，不構成投資建議。"

跨源打架收尾句（两种场景，按三格实际读数择一或改写，不是唯一措辞）：
- 情绪与资金背离（推特情緒火热、真金白銀没跟上）→「情緒滿格 ≠ 大錢進場」（spec 逐字句式）
- 三格互不指向同一方向（无法收敛成一个判断）→「三個溫度計指向不同方向時，別把任何一個當聖旨。」
结论必须诚实反映三格各自的实际读数，不能为了叙事流畅硬圆成一个方向（S-5 多空平衡镜像）。

若推特情緒或內部人動向行引用到具体价格、目标价或涨跌幅数字（如某 KOL 喊单帖提到的目标价、或内部人申报的交易金额），这些数字只能引用本次调用返回值，KOL 转述或新闻里的二手数字必须先经快照核实才能写——呼应 S-7 铁律 2（原文：铁律2 价格：当前价/涨跌幅只引本次 MCP 返回值；新闻与 KOL 转述的百分比是二手，必经快照核实）。

**只呈现温度、不给跟单指令**（frontmatter description 定位）：三格读数即使全部指向同一方向，也不能写出"所以应该跟上/建议加仓"这类结论句——skill 只负责客观呈现三个维度各自的读数，买卖判断留给读者自己；给买卖指令、给仓位建议是 S-6 禁令的红线（原文：不给买卖指令、不给仓位建议、不预测短线涨跌；禁用"必涨/起飞/上车/梭哈/抄底"），本 skill 尤其容易因为"三源共振"而忍不住多说一句，需格外注意收住结论。

字数：单篇 500-800 字（S-4 默认档，c4 未被列为例外）。Top 3-5 标的每个三格若导致超长，运营可只保留 Top 3 出稿，Top 4-5 留作候补下周再跑，不必为了凑够 5 个硬塞。

以下是结构示范样例（非实测数据，实现时按当次调用返回替换；排版规则不变——纯文字＋emoji，不套 markdown 加粗/标题/表格，层级靠 emoji ＋ 空行 ＋「｜」——S-9 镜像）：

```
📌 本週熱議溫度計｜7/20–7/26

一句話先懂：本週推特喊單最熱是 MU（9 帖全多），但真金白銀沒跟上，情緒溫度和實盤資金出現落差。

本週推特在吵什麼
喊單聚合榜：1️⃣ MU　2️⃣ SPCX　3️⃣ NBIS｜整體多空比 12:1，情緒偏熱

MU 溫度計
🐦 推特情緒：多 9：空 0｜正：算力型 KOL（A 級）「DRAM 漲價週期才剛開始」｜反：暫無明顯反方聲音，一面倒偏多本身就是風險訊號
💰 真金白銀：實盤僅 1 位交易員輕倉做多，槓桿 2x｜過濾無評級與空值後樣本仍偏薄，本週實盤數據樣本太薄，不足以下判斷
🏢 內部人動向：近 90 天僅 CEO 兩筆 S-Sale（賣出），無 P-Purchase

跨源打架：情緒滿格 ≠ 大錢進場——推特一面倒看多，但真金白銀和內部人動向都沒有同步跟上。

接下來看什麼：留意真金白銀下週是否回補，若持續背離，這波情緒面可能只是噪音。

今日名詞：真金白銀
交易員拿自己真實資金開的實盤倉位，和「喊單」不同——喊單可能只是嘴上看好，真金白銀才是拿錢投票。可以粗略理解成：聽人講不如看他自己敢不敢下注。

⚠️ 以上整理自公開研報與市場數據，僅做資訊分享，不構成投資建議。
```

## 3. 防坑镜像

spec 逐字 6 条防坑规则（详见 caveats SSOT §6 N 系列 + 已知问题登记）：

> 防坑规则（本次实测新增，详见 §6）：
> - kol_call 按提及展开，一条推文裂成多行：按 source_url 去重、按 symbol 字段归属（N-5）。
> - insider/congress 行无视 time_range，必须客户端按 transactionDate 过滤（N-6）。
> - 13F 申报季中期数据残缺，禁止引用环比（N-7）。
> - trader_position 过滤 tier 为空/notional null 的行；美股覆盖日级剧变，只能当天现拉、按 underlying 合并符号分裂（既有 caveat）。
> - stock_perp 是代币化股票永续，呈现时标注"鏈上股票永續交易員"而非"美股交易員"。
> - signal query 不放"KOL"等元词（会被解析成 crypto 关键词，实测空返）。

逐条展开：

- **kol_call 去重+归属（N-5）**：

  > N-5：kol_call 原帖按提及 fanout 成多行（同 URL 不同 symbol/方向）；按 source_url 去重、symbol 字段归属（实测 2026-07-22）。

  步骤 1/2 返回的 kol_call 类数据，同一条原始喊单推文可能因为提到多个标的被拆成多行（同一个 source_url，symbol 或方向字段不同）。总览段的喊单聚合榜排名、以及每个标的推特情緒行的多空比统计，都必须先按 source_url 去重，再按 symbol 字段把行归属回正确的标的——不去重会让同一条推文的热度被重复计入喊单榜排名，虚增某个标的的讨论热度。

- **insider/congress 客户端过滤 transactionDate，只认 S-Sale/P-Purchase（N-6）**：

  > N-6：insider/congress 行无视 time_range（7d 返回 2020 年记录）；客户端按 transactionDate 过滤强制（实测 2026-07-22）。

  內部人動向行必须客户端按 transactionDate 过滤——API 不管 time_range 传了多少，insider/congress 类的行照样可能混入多年前的旧记录。c4 的窗口是近 90 天（不同于 c1 早报镜像 N-6 时用的"昨日"单日窗口——本 skill 是周跑产品，要给一个有意义的近期趋势而不是单日快照，二者窗口不同是有意为之，不要混淆）。过滤后只认 S-Sale（卖出）、P-Purchase（买入）两种交易类型，其余类型一律不计入——既有 insider 条目的扩充解读规则说明了原因：F-InKind/M-Exempt 为缴税代扣非主动交易；对外表述"内部人卖出"只认 S-Sale，买入只认 P-Purchase。

- **13F 申报季禁环比，本模块加严为一律禁用（N-7）**：

  > N-7：13F institutional 申报季中期 investorsHolding 环比为残缺假信号（实测 NVDA 6234→1441）；申报季内禁止引用环比（实测 2026-07-22）。

  N-7 本身是条件禁令——只在申报季内才禁止引用 investorsHolding 等 13F 环比字段。本模块执行更严格的版本：不判断当下是否处于申报季，一律不引用任何 13F 环比/较上季变化的字段，产出只能引用当次返回的绝对持仓数字（如"目前 X 家机构持有"）——这是 N-7 的模块级执行，避免运营每次都要额外核实"现在是不是申报季"这个前提条件才敢下笔。

- **trader_position 当天现拉 + underlying 合并（既有 caveat）**：

  > trader_position 美股标的覆盖**日级剧变**（实测 07-09 MU 4 人 vs 07-15 MU 1 人、海力士从无到 3 人）；且同一标的可能符号分裂成多组（海力士 underlying=000660.KS 散在 SKHYNIX/SKHX/SKHY 三个 symbol）。Workaround：任何对外用途都当天现拉；空 keywords 拉 trending 看当前有货标的；符号分裂需按 underlying 合并（数据特性，非 bug）。

  真金白銀行统计前先过滤 tier 为空、notional 为 null 的行；美股标的的实盘持仓覆盖会逐日剧变，产出前必须当天现拉，不能沿用几天前甚至上周跑过的缓存数据；同一标的可能因符号分裂成多个 symbol（如海力士拆成 SKHYNIX/SKHX/SKHY），统计人数/方向/杠杆前要按 underlying 字段合并，否则会把同一批交易员的仓位重复计成好几个不同标的。

- **stock_perp 标注「鏈上股票永續交易員」**：

  > stock_perp 是代币化股票永续，呈现时标注"鏈上股票永續交易員"而非"美股交易員"。

  真金白銀行若命中的持仓类型是 stock_perp（代币化股票的链上永续合约，而非真实美股经纪商持仓），呈现时必须标注"鏈上股票永續交易員"，不能写成"美股交易員"——两者交易的是完全不同的资产，混为一谈会让新手误以为这是真实美股仓位。

- **signal query 禁放"KOL"等元词**：

  > signal query 不放"KOL"等元词（会被解析成 crypto 关键词，实测空返）。

  步骤 2 的 query 只能是"<TICKER> 详细仓位"这类标的+意图词组合，禁止把"KOL"这类元描述词塞进 query——会被服务端解析成加密货币关键词，实测直接空返，四维钻取全部落空却不会报错，容易被误判成"这个标的没有信号"而错误放弃该标的。

## 4. 发前自检 + 额度哨兵

S-3 骨架适配（去掉第 7 段互动钩子）：

> 只出现在 c1/c5、一天最多一次、问题无立场不诱导买卖方向、c3/c4 严肃内容不加。

温度计三格是严肃的跨源数据交叉呈现，S-3 原文明确点名 c4 属于"严肃内容不加"的例外——不加开放式讨论钩子。

S-12 清单（逐字，每篇产出前逐项过）：

> 繁体✓ 大白话✓ 字数✓ 名词卡✓ 免责✓ 多空平衡✓ 单源标注✓ 价格可回溯✓

c4 专属追加一项：**实盘数据当天现拉✓**——真金白銀行引用的 trader_position 数据是不是当次调用现拉的（不能沿用之前跑过的旧缓存），呼应第 3 节"日级剧变"的既有 caveat。

额度哨兵（内部提醒，不进对外贴文，设计文档对应章节）：

> **额度哨兵**：每次模块跑完读一次 `meta.quota`（每个返回都带，实测），`remaining/limit < 15%` 时在产出末尾附一行内部提醒（不进对外贴文）："⚠️ 本月額度剩 N 次，按當前節奏約可跑 X 天，考慮降頻或升級"。额度见底导致日更断档比任何单篇内容缺失都伤运营。
